# Frontend i18n Design (EN/中文)

## Goal

Add lightweight i18n support for all frontend UI copy with manual language toggle in the top-right, defaulting to English and persisted in localStorage. Remove the "Publish Skill" navigation item.

## Scope

- Frontend-only: all UI copy across pages.
- New i18n layer with dictionary + provider + hook.
- Manual language switcher in header, persists selection.
- Remove "Publish Skill" from navigation.

Out of scope:
- Backend changes
- Automatic language detection
- Advanced formatting (plural rules, dates)
- External i18n libraries

## Approach

Use a minimal in-house i18n implementation:
- `messages.ts`: dictionary for `en` + `zh` keyed by string ids.
- `I18nProvider`: context storing current locale and setter.
- `useI18n()`: returns `t(key)` and `locale`, `setLocale`.
- Default locale is `en`, overridden by `localStorage`.
- Any missing key returns the key itself.

## Architecture

```
App
  I18nProvider
    Layout (language switcher button)
      Routes
        HomePage
        RepositoryDetailPage
        DocsPage
```

All UI strings are replaced by `t("...")`.

## UX

- Top-right language switcher displays "EN / 中文".
- Default language is English.
- Selected language is persisted in `localStorage` (`mcp_market_locale`).

## Pages Covered

- Layout: navigation, footer, buttons, labels.
- HomePage: hero, search placeholders, empty states, CTA labels.
- RepositoryDetailPage: install flow labels, section titles, empty state, toast messages.
- DocsPage: all documentation copy.

## Error Handling

- Missing translation key → show the key as fallback.

## Testing

- `cd frontend && bunx tsc -b`
- `cd frontend && bun run build`
